# 文档整理流程 Prompt

本文档定义了一个通用的文档整理工作流程，适用于任何软件项目。当AI完成代码变更后，应执行此流程来更新和整理项目文档。

---

## 工作流程概述

本流程确保项目文档始终与代码变更保持同步，文档结构清晰合理，所有链接有效可用。

---

## 步骤1: 更新教程文档和功能说明

### 目标
确保所有教程文档和功能说明文档反映最新的代码变更。

### 执行步骤

#### 1.1 识别受影响的教程文档
- 查找所有教程文档（通常在 `docs/tutorials/` 或 `tutorials/` 目录）
- 识别与新功能相关的教程章节

#### 1.2 更新功能说明文档
```markdown
## 更新示例

在适当位置添加新功能说明：
- 使用 "⭐ 版本号" 标记新功能
- 提供详细的功能描述
- 包含配置参数说明
- 提供使用示例
- 添加性能提升数据

## 示例格式

### 新功能名称 ⭐ v版本号

**功能特性**:
- 特性1
- 特性2

**配置参数**:
```python
PARAM_NAME = value  # 说明
```

**使用示例**:
```bash
# 示例命令
command --option
```

**性能提升**:
- 指标1: 数据
- 指标2: 数据
```

#### 1.3 更新场景示例文档
- 添加新功能的使用场景
- 提供完整的示例代码
- 展示实际使用效果
- 包含最佳实践建议

#### 1.4 更新文档间的交叉引用
- 在相关功能文档中添加链接到新功能文档
- 在新功能文档中添加链接到相关文档

## 步骤5: 整理文档结构

### 目标
确保文档结构清晰合理，无重复文档。

### 执行步骤

#### 5.1 评估文档结构
```
## 建议的文档结构

docs/
├── README.md                    # 文档中心导航
├── tutorials/                   # 教程文档
│   ├── 01-overview.md
│   ├── 02-installation.md
│   ├── 03-scenarios.md
│   ├── 04-features.md
│   └── ...
├── design/                      # 设计文档
│   ├── DESIGN_SUMMARY.md
│   └── ...
├── implementation/              # 实现文档
│   ├── IMPLEMENTATION_GUIDE.md
│   └── ...
├── testing/                     # 测试文档
│   └── TEST_DESIGN.md
├── general/                     # 一般文档（保留重要文档）
│   ├── PROJECT_ROADMAP.md
│   ├── SECURITY_DOCUMENTATION.md
│   ├── USE_CASES.md
│   └── ...
├── history/                     # 历史文档（修复和实施总结）
│   ├── README.md
│   ├── FIX_REPORT.md
│   └── ...
├── future-feature-design/       # 未来功能设计
│   └── README.md
└── implemented-features/         # 已实现功能
    ├── f1-feature-name/
    ├── f2-feature-name/
    └── ...
```

#### 5.2 识别需要移动的文档
- 检查 `docs/general/` 目录
- 将历史修复报告移到 `docs/history/`
- 将临时文档移到 `docs/history/`
- 保留重要的长期文档在 `docs/general/`

#### 5.3 执行文档移动
```bash
# 示例命令
mkdir -p docs/history

# 移动历史文档
mv docs/general/AGENT_FIX_SUMMARY.md docs/history/
mv docs/general/OCR_CONFIG_FIX.md docs/history/
mv docs/general/IMAGE_HANDLING_FIX.md docs/history/
# ... 继续移动其他历史文档
```

#### 5.4 创建历史文档目录的README
```markdown
# 历史修复和实施文档

本目录包含项目历史修复报告和实施总结文档。

## 📜 历史文档列表

列出所有历史文档
```

## 步骤6: 检查和修复文档链接

### 目标
确保所有文档间的链接都是有效的。

### 执行步骤

#### 6.1 搜索所有Markdown链接
```bash
# 搜索所有Markdown文档中的链接
grep -r '\]\(.+\.md\)' docs/
```

#### 6.2 验证链接有效性
- 检查每个链接指向的文件是否存在
- 验证相对路径是否正确
- 确认锚点链接是否有效

#### 6.3 修复无效链接
```markdown
## 常见修复场景

### 场景1: 文件已移动
**修复前**: `[文档](general/DOCUMENT.md)`
**修复后**: `[文档](history/DOCUMENT.md)`

### 场景2: 文件名错误
**修复前**: `[文档](DOCUMNET.md)`  # 拼写错误
**修复后**: `[文档](DOCUMENT.md)`

### 场景3: 相对路径错误
**修复前**: `[文档](../../path/file.md)`
**修复后**: `[文档](../path/file.md)`
```

#### 6.4 更新导航链接
- 检查文档中心的导航链接
- 检查教程文档中的前后导航链接
- 检查文档中的交叉引用

#### 6.5 验证根目录链接
```bash
# 检查根目录是否有重复文档
find . -name "CHANGELOG.md" -not -path "./.git/*"
find . -name "README.md" -not -path "./.git/*"

# 删除重复文档
rm src/CHANGELOG.md  # 如果根目录已有
```

## 步骤7: 文档拆分或合并决策

### 目标
评估是否需要拆分或合并文档。

### 决策标准

#### 7.1 需要拆分文档的场景
- 单个文档超过1000行
- 单个文档包含多个不相关的主题
- 单个文档维护困难

#### 7.2 需要合并文档的场景
- 多个文档内容高度重复
- 多个文档描述同一功能的不同阶段
- 单个文档内容过少（少于50行）

#### 7.3 执行拆分
```bash
# 示例：拆分大文档
# 原始文档: large-document.md

# 拆分为:
# - large-document-part1.md
# - large-document-part2.md
# - large-document-summary.md (目录)
```

#### 7.4 执行合并
```bash
# 示例：合并相似文档
# 原始文档:
# - fix-v1.md (第一次修复)
# - fix-v2.md (完整修复)

# 合并为:
# - fix-complete.md (保留最完整的版本)
# 删除: fix-v1.md
```

## 最佳实践

### 文档命名约定
- 使用小写字母和连字符: `feature-name.md`
- 使用描述性名称: `file-validator.md` 而不是 `module.md`
- 使用版本标识: `v1.0.0-implementation.md`

### 文档组织原则
- 按功能组织：相关的文档放在同一目录
- 按时间组织：历史文档放在 history/ 目录
- 按类型组织：设计、实现、测试分开存放

### 链接管理原则
- 使用相对路径：`../path/file.md` 而不是绝对路径
- 避免硬编码路径
- 保持链接的上下文相关性
- 定期检查链接有效性

### 版本控制原则
- 每次代码变更都要更新文档
- 在文档中明确标注版本号
- 保留历史文档以供参考
- 删除完全过时的文档

## 常见问题和解决方案

### 问题1: 文档更新遗漏
**原因**: 没有识别所有受影响的文档
**解决**: 使用 `find` 和 `grep` 搜索相关关键词

### 问题2: 链接失效
**原因**: 文件移动后未更新链接
**解决**: 移动文件后立即更新所有引用

### 问题3: 文档结构混乱
**原因**: 没有遵循组织原则
**解决**: 参考本文档的建议结构重新组织

### 问题4: 版本不一致
**原因**: 多处版本号未同步更新
**解决**: 使用全局搜索替换统一更新
